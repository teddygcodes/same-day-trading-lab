# Build Prompt — v0.5a: Combinatorial strategy enumerator + frozen vocabulary

**For the coding session.** Read `CLAUDE.md` first — especially the new
**Combinatorial discovery (v0.5)** section, which sanctions this work *only* under the
empirical-null + gauntlet discipline. This is **step 1 of 3** (v0.5a/b/c). Build only the
enumerator + vocabulary here; **the null, E_FP/S-N, baselines, and the three-window
gauntlet are v0.5b/v0.5c — do NOT build them now.** Reuse the replay/fill/friction engine
**unchanged**.

## Mission

Replace hand-typed strategy classes with a deterministic **enumerator** that composes
candidate strategies from a **frozen, versioned, pre-registered vocabulary** of orthogonal
pieces. Each candidate emits the existing generic `TradePlan` and is runnable through the
current `run` / `run-range` / `tournament` paths by a stable id. The three existing
strategies become vocabulary pieces; the truth engine underneath does not change.

## Vocabulary v1 — intraday-only, engine-expressible (this is the load-bearing scope call)

Every piece is a **pure function over a `ReplayView` of completed bars** (the no-lookahead
firewall applies identically to every generated candidate). v1 includes **only** pieces
that (a) use *current-day* completed bars and (b) are priceable by the existing fill engine
via `TradePlan(trigger, stop, target_r_multiple)`. Define a concrete, orthogonal v1 set,
e.g.:

- **Entry triggers** (the existing three + a few intraday peers): `orb_breakout`,
  `vwap_reclaim`, `or_fade`, `intraday_high_break`, … (choose a fixed set; all long-only,
  all close-based, all from completed bars).
- **Context filters** (boolean, evaluated at the trigger bar on completed bars): e.g.
  `time_of_day_window`, `trend_vwap_alignment`, `or_width_sane`, `prior_bar_up`, … (a fixed
  set of independent toggles; each candidate carries a frozen subset).
- **Exits**: a small set of **R-multiples** (e.g. `1R`, `2R`, `3R`) expressed via
  `target_r_multiple`; flatten remains at session close (unchanged engine).
- **Stops**: `or_low`, `intraday_swing_low`, `atr_intraday` (ATR from completed bars).

**Enumerate the FULL Cartesian product. Do NOT prune "nonsensical" combinations** — quality
pruning is a hidden optimizer (CLAUDE.md). The exact candidate count is whatever the v1 set
yields; it need not match any preconceived number.

### Explicitly DEFERRED (do not add in v0.5a)
Pieces needing **cross-day context** (`prior_day_high_break`, `relative_volume_gate`) or
**dynamic exits** the engine can't price (`vwap_touch`, `trailing`, `time_stop`). These
require a separate context-provision and/or fill-engine extension and a **vocabulary v2**.
Note them in a `# DEFERRED (vocabulary v2)` comment so the intent is recorded; building them
now would either break "engine unchanged" or violate per-day independence.

## What to build

- **`same_day_lab/strategy/vocabulary.py`** (or similar): the v1 piece definitions + a
  `VOCABULARY_VERSION` string. Pieces are small pure callables grouped by kind.
- **An enumerator** that produces the full Cartesian product as candidate specs
  `(trigger, frozenset(filters), exit, stop)`, each with a **deterministic stable id**
  (hash of its piece names + `VOCABULARY_VERSION`).
- **A composed `GeneratedStrategy`**: `on_bar(view) -> TradePlan | None` — one trade/day;
  fires when the trigger fires **and all filters pass** (on completed bars); emits
  `TradePlan(trigger_price=close, stop_price=<stop rule>, target_r_multiple=<exit R>)`.
  Reads only the `ReplayView`. Refactor the existing 3 strategies to be expressible as
  vocabulary pieces (their current behavior must remain reproducible).
- **Registry integration**: `get_strategy(id)` resolves a generated candidate id to a
  `GeneratedStrategy`, so `run`/`run-range`/`tournament` can run any candidate by id.
- **`same_day_lab/cli.py` → `generate-strategies`**: emit/verify the committed candidate
  list to a file (e.g. `config/vocabulary_v1_candidates.json`) with `vocabulary_version`,
  the piece sets, and every candidate id. CLI help (and a code comment) must state:
  **editing the vocabulary after viewing any results invalidates pre-registration — bump
  the version.**

## Tests (offline, TDD)
- **Enumerator determinism:** for `VOCABULARY_VERSION`, the enumerator reproduces exactly
  the committed candidate list and ids (hash match); two runs identical.
- **Firewall holds on a composed candidate:** the poison-future-bar and same-bar-fill
  guards still raise for a `GeneratedStrategy` (reuse the existing firewall tests' shape).
- **Pieces are pure** (no I/O); filters/triggers/stops evaluated only on completed bars.
- **End-to-end:** a chosen generated candidate runs on a bundled fixture and emits a valid
  `TradePlan`; the three refactored existing strategies reproduce their prior behavior.
- Full suite green, offline.
