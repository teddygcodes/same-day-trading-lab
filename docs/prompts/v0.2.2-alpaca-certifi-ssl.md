# Build Prompt — v0.2.2: Alpaca client uses certifi so live pulls work out of the box

**For the coding session.** Read `CLAUDE.md` first. Small, surgical fix. Do exactly this.

## The problem

A live Alpaca pull (`tools/record_real_day.py` / `ingest --provider alpaca`) fails on a
python.org macOS build with `[SSL: CERTIFICATE_VERIFY_FAILED] unable to get local issuer
certificate` — the stdlib `ssl` module has no CA bundle. TCP egress works; only TLS
verification fails. Today it only succeeds if the user manually sets `SSL_CERT_FILE`.

## The fix

Make the Alpaca client verify TLS against **certifi's** CA bundle when certifi is
importable, falling back to the stdlib default context otherwise. Keep verification
**ON** — this is a financial API.

- In `same_day_lab/ingest/alpaca.py`, add a small helper, e.g.:
  ```python
  def _ssl_context():
      ctx = ssl.create_default_context()
      try:
          import certifi
          ctx.load_verify_locations(certifi.where())
      except Exception:
          pass  # fall back to system defaults
      return ctx
  ```
  and pass `context=_ssl_context()` to `urllib.request.urlopen(...)` in `_get_json`.
- **Dependency:** add `certifi` as an **optional extra**, not a core dep (keep the core
  stdlib-minimal per CLAUDE.md):
  ```toml
  [project.optional-dependencies]
  alpaca = ["certifi"]
  ```
  So `pip install -e ".[alpaca]"` enables verified live pulls; the fixture/offline path
  needs nothing new. This build prompt is the sanction CLAUDE.md requires for a new dep.

## Do NOT
- **Never** disable verification: no `ssl._create_unverified_context`, no
  `check_hostname=False`, no `CERT_NONE`. The fix must keep certs verified.
- Don't touch anything beyond `alpaca.py` + `pyproject.toml` + a small test.
- Don't edit `README.md` / `CLAUDE.md` — report doc-impacting notes (the new `[alpaca]`
  extra) back to the planning agent.

## Tests (offline, no network)
- Unit-test `_ssl_context()` returns an `ssl.SSLContext` with `verify_mode == CERT_REQUIRED`
  and `check_hostname is True` (verification stays on).
- The existing stubbed-`urlopen` Alpaca tests must still pass (they monkeypatch
  `urlopen`, so the `context=` kwarg is harmless). If a test asserts call args, update it
  to tolerate/expect the `context` kwarg.
- Do not attempt a real network call in the suite.

## Acceptance

- `alpaca.py` passes a certifi-backed SSL context to `urlopen` when certifi is importable,
