# Build Prompt — v0.3: Multi-day replay + aggregate (per-day independent)

**For the coding session.** Read `CLAUDE.md` first; its prohibitions and invariants
bind. Build the smallest working spike for the scope below — nothing more.

## Mission

Run the **same symbol and the same strategy across a date range**, each day replayed in
**full isolation**, and aggregate the **distribution** of outcomes. v0.1/v0.2 answered
the fill-honesty question for one day; v0.3 asks whether the friction-death pattern is
**consistent across many days** — the statistical weight behind the thesis.

The narrow question: *Across N independent days, how often does an apparently-profitable
(naive) ORB-long trade survive pessimistic fills + the friction threshold — and how often
does friction kill it?*

## NON-NEGOTIABLE guardrails (this is where multi-day could drift into a backtester)

1. **Per-day independence is absolute.** Each day gets its own `ReplayClock` over that
   day's RTH bars, its own opening range, its own one-trade-per-day. **No** position,
   capital, or state carries between days. **No compounding.** P&L is per-share, per-day.
2. **This is NOT a portfolio backtester.** No cumulative capital, no position sizing, no
   equity curve, no drawdown, and **no Sharpe/Sortino/annualized** anything (CLAUDE.md
   prohibition). The aggregate is a **descriptive distribution / counts**, not a track
   record.
3. **No aggregate PASS/validation verdict.** Per-day verdicts stand as-is. The range
   output is descriptive ("X of Y traded days survive friction; naive overstated edge on
   Z") — it must NOT emit a single PASS that implies the strategy is validated.
4. **No-lookahead stays structural**, reusing the existing firewall unchanged; per-day
   isolation makes cross-day lookahead impossible too. Don't add a market-calendar
   dependency — operate over the days that are actually **ingested**, don't enumerate
   holidays.
5. **Data honesty:** never fabricate or silently skip. A range date with no ingest is
   reported as **missing** (told to record/ingest it); an `INVALID_DATA` day is counted
   as such. The **DATA WARNING** is the first block of the aggregate report.
6. **Determinism:** an `aggregate_hash` over the analytical core (the per-day cores +
   the aggregation) excludes volatile provenance, exactly like `report_hash`. Same
   fixtures + config ⇒ same `aggregate_hash`.

## Scope

### 1. Reuse, don't duplicate (small refactor in `cli.py`)
Extract the per-day pipeline currently inside `_cmd_run` into a reusable
`run_one_day(conn, symbol, date, config) -> dict` (returns the per-day report dict /
analytics) and have **both** `run` and the new `run-range` call it. Do not re-implement
fill/verdict logic.

### 2. New CLI: `run-range`
`same-day-lab run-range --symbol AAPL --start YYYY-MM-DD --end YYYY-MM-DD [--db]`
- Find the **ingested** `(symbol, date)` rows with `start <= date <= end` (query the DB).
- For each, call `run_one_day(...)`; collect per-day analytics.
- Range dates with no ingest → list them as **missing** in the output (do not invent).
- Per-day DB rows (`runs`/`trades`) are written as usual; individual per-day report
  *files* are optional (the aggregate is the deliverable).

### 3. Aggregate report (JSON + MD), written to `reports/`
- **DATA WARNING** first.
- **Per-day table:** date, data_valid, verdict, traded?(bool), exit_reason, naive_pnl,
  pessimistic_default_pnl, pessimistic_pass_pnl, crossover_cents.
- **Aggregate counts:** days ingested; # INVALID_DATA; # no-signal; # traded. Among
  traded days: # naive_pnl > 0, # pessimistic_default > 0, # surviving the pass-threshold,
  # KILL (naive>0 & pessimistic_default<=0). Distribution of `crossover_cents`.
- **Fill-honesty headline:** count of days where naive looked profitable but pessimistic
  killed it — the thesis evidence.
- **No aggregate verdict.** A one-line descriptive disposition is fine; a PASS is not.
- `aggregate_hash = content_hash(core)` over the deterministic core only.

### 4. Optional thin range-recorder (`tools/record_real_range.py`, creds-gated, untested)
Loops `record_real_day` over weekdays in a range; a day Alpaca returns empty for (holiday/
non-trading) is **skipped with a note**, not an error. Keep it minimal and out of the
test suite (like `record_real_day.py`). If this balloons, drop it and document a manual
loop instead — the testable deliverable is `run-range` + the aggregate.

## Tests (offline, TDD)
Add a few small multi-day synthetic fixtures (extend `tools/gen_fixture.py`) covering:
a day that trades and survives friction, a day that trades and is KILLed (naive>0,
pessimistic<=0), a no-signal day, and an INVALID_DATA day. Then:
- `run-range` aggregate counts are correct for that mix.
- **Determinism:** two range runs → identical `aggregate_hash`.
- **Per-day independence:** the result of day N is unchanged regardless of day N-1's
  outcome (e.g., a stop-out day before a trade day doesn't alter the trade day's OR/fills).
- A range date with no ingest is reported **missing**, not fabricated.
- Existing suite stays green.

## Do NOT
- Portfolio/capital/compounding/equity-curve/drawdown/Sharpe/Sortino/annualized.
- Cross-day state, a market-calendar dependency, or silently skipping missing days.
- An aggregate PASS/validation verdict.
- Multi-symbol (still one symbol per range run).
- Edit `README.md` / `CLAUDE.md` — report doc-impacting notes (new command, aggregate
  report shape, any new table) back to the planning agent.

## Acceptance criteria
- `pip install -e ".[test]" && pytest` green, fully offline.
- `run-range` over ingested days produces a deterministic aggregate JSON+MD with the
  DATA WARNING, the per-day table, the distribution counts, and the fill-honesty headline
  — and **no** aggregate validation verdict.
- Per-day independence and `aggregate_hash` determinism are proven by tests. Missing
  range dates are surfaced, never invented. No prohibited surfaces added.
