# Build Prompt — v0.2.1: Fixture loader resolves recorded real days (no silent substitution)

**For the coding session.** Read `CLAUDE.md` first; its invariants bind. This is a
small, surgical fix — do exactly this and nothing more.

## The bug

`tools/record_real_day.py` writes a recorded real session to
`fixtures/real_<symbol.lower()>_<date>_iex.json`. But `same_day_lab/ingest/fixture.py`
`load_fixture(symbol, date)` resolves only three hardcoded `(symbol, date)` pairs and
**silently falls back to `sample_aapl_1m_day.json`** for anything else.

So the documented real-pull flow —
`same-day-lab ingest --provider fixture --symbol AAPL --date <recorded day>` — does
**not** load the recorded real file. It silently serves the synthetic sample day under
the wrong date (or causes a confusing `run` "ingest first" failure because the sample's
`session_date` differs). **Silently substituting fabricated/sample data for a requested
real date violates the lab's provenance and honesty stance** (CLAUDE.md invariant 7).

## The fix (`same_day_lab/ingest/fixture.py` only)

Make `load_fixture(symbol, date, fixtures_dir=None)` resolve in this order:

1. **Known fixture map** — if `(symbol, date)` is in `_FIXTURE_FILES`, use it. (These
   three synthetic dates — 2025-05-15, 2025-11-28, 2025-06-16 — stay reserved and take
   precedence; real pulls must use other dates.)
2. **Recorded-real convention** — else if `real_<symbol.lower()>_<date>_iex.json` exists
   in the fixtures dir, use it. (Match `record_real_day._default_out` exactly.)
3. **Otherwise raise** a clear `FileNotFoundError` (or a small dedicated exception) —
   e.g. *"no fixture for {symbol} {date}; record one with tools/record_real_day.py or
   use a known fixture date"*. **Never silently fall back to the sample day.**

Notes:
- Recorded real files carry their own matching `session_date`, so the existing
  date-mismatch `fixture_notes` branch becomes effectively dead for them; you may leave
  it or remove it — don't add new behavior.
- A filesystem existence check is fine here (this is the offline provider loader, not a
  pure quality function).
- The `ingest` CLI should surface the raised error gracefully (non-zero exit, no
  traceback) the same way it already handles `AlpacaCredentialsMissing`.
