# Build Prompt — v0.4b: The honest tournament (leaderboard + holdout + multiple-comparisons)

**For the coding session.** Read `CLAUDE.md` first; its prohibitions/invariants bind.
This is **step 2 of 2** (v0.4a — the strategy abstraction + the three registered strategies
— is already done and on `main`). Build the smallest working spike for this scope.

## Mission

Evaluate the **full pre-registered strategy set** (the registry from v0.4a) over a
**decide** window and a **holdout** window, and emit a **descriptive leaderboard** with the
honesty machinery that makes multi-strategy testing legitimate. The whole point is to
*resist* over-reading promising in-sample results — so the output makes belief HARD, not
easy.

## Why this exists (do not undermine it)

Testing several strategies over the same history and keeping the best-looking one is
selection bias / p-hacking — the exact self-deception this lab exists to prevent. The two
guards:
- **Holdout:** decide on one window, then *confirm once* on a separate, unseen window. A
  strategy is only "carried forward" if it survives in **both**.
- **Multiple-comparisons honesty:** the report states how many strategies were tried, so a
  single decide-window PASS is flagged as weak evidence (no p-values — honest counting +
  the holdout gate).

## Guardrails (hold the line)
- Evaluate the **entire registered set** (no `--strategies` cherry-pick flag — choosing the
  subset after peeking would be the bias we're guarding against).
- **No optimization / parameter search**, no live loops, no LLM, no portfolio/compounding,
  long-only (all inherited from the strategies; don't add new ones here).
- **No tournament "winner" / validation verdict.** Per-strategy, per-day verdicts stand;
  the tournament is descriptive. A strategy is at most "corroborated (survived both
  windows)" — never "validated."
- Reuse v0.3's `run_range` + `reports/aggregate.py` per (strategy, window); the tournament
  is a thin cross-strategy × cross-window rollup. Don't fork the fill/verdict logic.
- Deterministic: a `tournament_hash` over the core (per (strategy, window) `aggregate_hash`
  values + the registered set + the two window identities), volatile provenance excluded.

## Scope

### CLI: `tournament`
`same-day-lab tournament --symbol AAPL --decide-start D1 --decide-end D2
--holdout-start H1 --holdout-end H2 [--db]`
- For each registered strategy and each window, compute the per-strategy aggregate
  (reuse `run_range`'s machinery; you may compute the aggregates **in memory** — you do not
  need to write each sub-aggregate file).
- Operate over **ingested** days only (same data-honesty as `run-range`): surface missing
  weekdays per window; never fabricate.

### Tournament report (JSON + MD), to `reports/`
- **DATA WARNING** first.
- **Leaderboard:** one row per strategy with the **decide** and **holdout** summaries side
  by side — for each window: days traded, # surviving the pass threshold, # KILL, and the
  fill-honesty count. Clearly label decide (in-sample, where we looked) vs holdout
  (confirmatory).
- **Corroboration flag:** per strategy, "carried forward" only if it survives in **both**
  windows (define "survives a window" explicitly, e.g. ≥1 day `PASS_FOR_MORE_TESTING` and
  net-non-losing — keep it simple and state the rule). This is a flag, not a verdict.
- **Multiple-comparisons caveat (prominent):** "N strategies evaluated; with N tries a
  single in-sample PASS is weak — corroboration on the holdout is required, and even that
  is suggestive, not proof." State N explicitly.
- **No aggregate/tournament verdict.** A one-line descriptive disposition is fine.
- `tournament_hash` over the deterministic core.
