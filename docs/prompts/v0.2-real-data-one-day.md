# Build Prompt — v0.2: Real Data, One Day

**For the coding session.** This is a self-contained task spec. **First read `CLAUDE.md`** —
every invariant and prohibition there is binding. This prompt only adds the v0.2 scope.

## Read first
Build the smallest working spike. Do not exceed this scope. v0.2 stays **one symbol,
one day, one strategy** — no multi-day, no second strategy, no engine, no live orders,
no async/threading/UI. When in doubt, do less. Stop when the acceptance criteria pass.

## Mission

Make the lab **ready to run on real Alpaca IEX data** for a single real session, and
**harden quality checks for real-world messiness** — without breaking the offline test
suite and without fabricating data.

The narrow question v0.2 answers: *Can the v0.1 harness ingest and replay a real (not
synthetic) 1-minute session — surviving the gaps, halts, and IEX-subset reality of
real data — and still produce an honest, deterministic verdict?*

This is still a **fill-honesty lab**, not a data-collection project and not a backtester.

## Hard reality this task must respect

1. **You (the coding session) cannot pull real data** — no Alpaca creds, no network in
   your sandbox, and the test suite must stay offline. So your job is to **build and
   prove the capability against a crafted "messy real-shaped" offline fixture**. The
   actual real pull is a follow-on manual step the user runs with creds (see Record tool).
2. **IEX bars have legitimate gaps.** On the free/IEX feed, a minute with no IEX trades
   produces **no bar**. A real RTH day will therefore have missing minutes. v0.1's
   `quality.max_missing_bars: 0` would mark essentially every real day `INVALID_DATA` —
   that is the central problem to fix.
3. **Never fabricate bars.** Do NOT forward-fill or synthesize bars to paper over gaps.
   Missing data is real information loss; surface it honestly, don't hide it.

## Scope

### 1. Harden the Alpaca client (`same_day_lab/ingest/alpaca.py`)
- Confirm/finish the real v2 request: `GET .../v2/stocks/{symbol}/bars?timeframe=1Min&
  start=&end=&feed=iex&limit=10000&adjustment=raw`. Use `adjustment=raw` (no
  corporate-action logic in v0.2). Keep pagination via `next_page_token`.
- Map errors to clear, graceful messages (no tracebacks): 401/403 → bad/missing creds;
  429 → rate limited (a single synchronous `time.sleep`-based retry honoring
  `Retry-After` is acceptable — **no async/threading**); empty `bars` → "no bars
  returned (non-trading day or bad symbol/date)".
- Stay stdlib-only (`urllib`). The live call remains untested offline by design.

### 2. Record-and-replay tool (`tools/record_real_day.py`)
- A small, creds-gated script: `python tools/record_real_day.py --symbol AAPL --date
  YYYY-MM-DD [--out fixtures/real_<symbol>_<date>_iex.json]`. It calls
  `alpaca.fetch_bars` and writes the **fixture-shaped** payload (same keys as existing
  fixtures; no creds in the file). This is how a real day becomes a committed, offline,
  reproducible regression fixture. Not part of the test suite.

### 3. Quality hardening (`same_day_lab/quality/`, `config/default.yaml`)
Implement a nuanced missing-bar policy and one new flag. Recommended config shape
(reconcile with existing keys; update `config_hash`-dependent tests):
```yaml
quality:
  # replaces the blunt max_missing_bars: 0
  missing_bar_policy:
    max_missing_fatal: 30          # > this many missing RTH minutes -> INVALID_DATA
    halt_run_min_consecutive: 10   # >= N consecutive missing minutes -> halt_suspected
```
- **Missing minutes are non-fatal up to `max_missing_fatal`**, but always **surfaced
  prominently** in the quality summary and report (count + the missing timestamps).
  Above the threshold → `INVALID_DATA` (real gap / bad data).
- **Add `halt_suspected`** (non-fatal): a run of `>= halt_run_min_consecutive`
  consecutive missing RTH minutes. Record it in the summary; do not invalidate.
- **Keep fatal:** suspicious OHLC, duplicate bars, partial session (these stay
  `INVALID_DATA`). **Keep DEFERRED:** `split_suspected` / `adjusted_suspected` (needs
  multi-day prior-close context — not now).
- Quality flag functions stay **pure**.

### 4. "Messy real-shaped" offline fixture (`tools/gen_fixture.py`, `fixtures/`)
- Extend the generator to emit `fixtures/messy_real_aapl_1m.json`: a full RTH day that
  resembles real IEX data — includes scattered legitimately-missing minutes (gaps), one
  mid-session halt-like run of consecutive missing minutes (≥ the halt threshold), an
  extreme-move bar, and a zero-volume bar. Keep it deterministic and OHLC-valid.
- It should produce a runnable trade so the end-to-end report still exercises fills.

## Tests (mandatory, offline, TDD)
- **Quality:** missing minutes below `max_missing_fatal` → `data_valid` True with a
  surfaced count; above → `data_valid` False. `halt_suspected` fires on a long missing
  run and is **non-fatal**. Assert **no fabricated bars** are introduced anywhere.
- **End-to-end on the messy fixture:** ingest + run produces a deterministic report +
  verdict (NOT auto-`INVALID_DATA` from expected IEX gaps); missing-bar accounting is
  present in the report.
- **Alpaca client (stubbed, no network):** monkeypatch `urllib.request.urlopen` to
  return a fake 2-page response and assert correct URL/params, pagination assembly, and
  error mapping (401/403/429/empty). 
- **Record tool (offline):** monkeypatch `alpaca.fetch_bars` and assert the tool writes
  a valid fixture-shaped file.
- Existing tests must stay green (update any that depend on the old `max_missing_bars`
  key or `config_hash`).

## Do NOT
- Pull real data, require network in tests, or commit creds.
- Fabricate/forward-fill bars to hide gaps.
- Add multi-day, a second strategy, an engine, live orders, async/threading, or UI.
- Implement split/adjustment detection (deferred).
- **Edit `README.md` or `CLAUDE.md`.** Those are owned by the planning agent. Instead,
  at the end, **report a short list of doc-impacting changes** (new config keys, new
  command/tool, changed quality semantics) so the planning agent can update the docs.

## Acceptance criteria
- `pip install -e ".[test]" && pytest` is green and fully offline.
- Alpaca client builds the correct request, paginates, and maps errors gracefully
  (proven with stubs, no network).
- `tools/record_real_day.py` exists and is documented as the creds-gated way to capture
  a real day into a fixture.
- The messy real-shaped fixture replays end-to-end into a deterministic report + verdict;
  IEX-style gaps are non-fatal and prominently surfaced; `halt_suspected` is detected
  and non-fatal; structural issues remain fatal; **no fabricated bars**.
- No new prohibited surfaces. `README.md`/`CLAUDE.md` untouched; doc-impacting changes
  reported back.

## Verification (do this before declaring done)
1. `pytest -v` — all green, no network.
2. `python tools/gen_fixture.py` then `same-day-lab init-db && same-day-lab ingest
   --provider fixture --symbol AAPL --date <messy-fixture-date> && same-day-lab run ...`
   → a report whose quality summary shows the gap/halt accounting and a sensible verdict.
3. Grep to confirm no `asyncio`/`threading`/`websocket`/UI/live-order code was added.
4. Confirm two runs of the messy fixture produce an identical `report_hash`.

## Handoff note
The real-day pull is a **separate manual step** the user performs with creds after this
lands:
```bash
export ALPACA_API_KEY=... ALPACA_SECRET_KEY=...
python tools/record_real_day.py --symbol AAPL --date <a real recent trading day>
same-day-lab ingest --provider fixture --symbol AAPL --date <that day>   # replays the recorded fixture offline
same-day-lab run --symbol AAPL --date <that day>
```
Report doc-impacting changes to the planning agent rather than editing README/CLAUDE.md.
